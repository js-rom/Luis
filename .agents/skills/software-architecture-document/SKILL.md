## Purpose

Create or refine the SW Architecture Document for the current iteration.

Architecture MUST be treated as the set of significant design decisions about system organization, structural elements, interfaces, collaborations, constraints, and evolution.

The document MUST stay at architecture level (packages/components/processes/deployment), not class-level implementation details.

## Ownership Boundary

This skill owns final document assembly and all documentation duties for:

- Architecture Overview section.
- 4+1 Views section.
- Traceability matrix and final consistency checks.

`architectural-design` provides decision inputs, but this skill owns the final documented structure.

## Required Inputs

- Vision and Business Case.
- Supplementary Specification (especially non-functional constraints).
- Use Case Model and selected iteration use cases/scenarios.
- Domain Model.
- SSDs and Operation Contracts for architecturally significant scenarios.
- Existing SW Architecture Document (if it exists).
- Current technology constraints, platform constraints, legacy constraints, and team/resource constraints.
- Architecture Decision Package generated by `architectural-design` (recommended input).

## Mandatory Pair Programming Workflow

The agent MUST execute this flow in order.

1. Assemble required inputs and open a document skeleton.
2. Integrate architecture drivers, scope, and significant decisions.
3. Document architecture overview (elements, connectors, boundaries).
4. Document architecture using 4+1 views.
5. Integrate style/pattern decisions, MVC section, and dependency governance.
6. Integrate quality attributes/scenarios, risks, and evolution plan.
7. Build traceability matrix and run full validation checklist.
8. Persist final document in the required storage path.

### Execution Mode (MANDATORY)

- Agent role: Driver.
- User role: Navigator.
- The agent proposes each section incrementally and requests feedback before moving to the next section.

## Architecture Rules (MUST / MUST NOT)

- MUST represent architecture using elements, externally visible properties, and relations between elements.
- MUST model three architecture element types: processing elements, data elements, and connectors.
- MUST describe interfaces and collaborations among components/subsystems.
- MUST separate logical layers from physical tiers.
- MUST keep traceability to use cases and quality attributes.
- MUST include constraints from non-functional requirements, technology, reusability, economics, and changeability.
- MUST include rationale for each significant decision.
- SHOULD use PlantUML diagrams for all architecture views.
- MUST NOT reduce architecture to class-level design patterns.
- MUST NOT include algorithmic implementation details.

## 4+1 Views Contract (MUST)

The document MUST include all five views.

1. Use Case View
- Captures architecturally significant use cases/scenarios that drive the architecture.
- Includes behavioral diagrams (use case and sequence/activity/state as needed).

2. Logical (Design) View
- Captures packages, interfaces, and collaborations that solve functional requirements.
- Focuses on software architecture structure.

3. Development (Implementation) View
- Captures components/modules and assembly/versioning boundaries.
- Focuses on build and configuration structure.

4. Process View
- Captures concurrency, process/thread model, synchronization, and runtime behavior.
- Covers operational, adaptability, and performance concerns.

5. Physical (Deployment) View
- Captures nodes/topology and deployment mapping.
- Covers distribution, installation, and runtime placement constraints.

For EACH view, include:

- Intent
- Scope
- Main diagram(s)
- Key constraints/rules
- Decisions and rationale

## Package Principles and Modularity Rules (MUST)

The document MUST explicitly apply and assess:

- Acyclic Dependencies Principle: package dependency graph must be acyclic.
- Stable Dependencies Principle: dependencies should point toward stability.
- Stable Abstractions Principle: most stable packages should be most abstract.
- Common Reuse Principle: classes reused together should be packaged together.
- Common Closure Principle: classes that change together should be packaged together.
- Release/Reuse Equivalency Principle: reuse granularity equals release/version granularity.

The document MUST provide dependency governance:

- Allowed dependency directions.
- Forbidden dependencies.
- Cycle breaking strategy.
- Coupling impact notes (efferent/afferent, at least qualitative; quantitative when available).

## Style and Pattern Selection Rules (MUST)

The document MUST evaluate style decisions against context and quality goals.

At minimum, evaluate relevant options from:

- Structural: Layered architecture.
- Persistence: Active Record, Row Data Gateway, Table Data Gateway/DAO, Data Mapper/ORM.
- Adaptable: Microkernel.
- Interaction:
	- Model/View/Controller
	- Model/View/Presenter with Presentation Model
	- Model/View/ViewModel
	- Model/View/Presenter with Passive View
	- Model/View/Presenter with Supervising Controller

For each considered style include:

- Context fit
- Constraints imposed
- Quality attributes improved/degraded
- Why selected or rejected

The document MUST distinguish:

- Architectural pattern/style (system level)
- Design pattern (subsystem/class level)

## MVC Presentation Architecture Contract (MUST)

The document MUST include one dedicated section titled `MVC and Presentation Architecture` with the following subsections in this exact order:

1. Model/View/Controller
2. Evolution
3. Model/View/Presenter with Presentation Model
4. Model/View/ViewModel
5. Model/View/Presenter with Passive View
6. Model/View/Presenter with Supervising Controller
7. Summary

Content rules per subsection:

1. Model/View/Controller
- MUST define responsibilities for Model, View, and Controller.
- MUST describe collaboration flow from user interaction to model update.
- MUST preserve separation between domain logic and presentation concerns.

2. Evolution
- MUST explain why classical MVC evolves in UI frameworks with native event controls.
- MUST describe the testability issue when View owns state/logic/synchronization.
- MUST explain what presentation concerns move toward a Presenter and why.

3. Model/View/Presenter with Presentation Model
- MUST state decoupling goal between Presenter and View.
- MUST assign full presentation state and logic to Presenter.
- MUST keep View focused on controls and minimal presentation synchronization.
- MUST include expected unit testing strategy for Presenter.

4. Model/View/ViewModel
- MUST state motivation to reduce synchronization coding.
- MUST describe automatic data binding role and boundaries.
- MUST declare whether the selected tech stack supports binding natively.

5. Model/View/Presenter with Passive View
- MUST reduce View to minimal interface-control responsibilities.
- MUST assign state/logic/synchronization to Presenter.
- MUST include how Presenter tests use a View substitute (mock/stub/fake).

6. Model/View/Presenter with Supervising Controller
- MUST split responsibilities as: View handles simple/immediate presentation tasks, Presenter handles complex tasks.
- MUST define explicit criteria for "simple" vs "complex" tasks.

7. Summary
- MUST provide a comparative summary matrix of all evaluated MVC-family variants.
- MUST justify the selected variant with quality, maintainability, and testability trade-offs.
- MUST include one explicit rejection reason for each non-selected variant.

Evidence rules:

- MUST include at least one interaction/sequence diagram for the selected presentation variant.
- MUST keep nomenclature consistent across sections (Model, View, Controller/Presenter, ViewModel).

## Quality Attribute Rules (MUST)

Prioritize and document quality attributes in three groups:

1. Quality attributes
- Portability
- Changeability/adaptability
- Understandability
- Testability
- Usability
- Security
- Data integrity

2. Operational attributes
- Availability
- Hot update capability
- Reliability
- Recoverability
- Manageability/observability

3. Performance attributes
- Response time
- Elasticity
- Throughput/capacity

For top-priority attributes, define measurable quality scenarios:

- Source
- Stimulus
- Environment
- Artifact
- Response
- Response measure

## Output Contract (MUST)

The produced artifact MUST include these sections in order:

1. Scope and Architecture Drivers
2. Significant Decisions and Rationale
3. Architecture Overview (elements, connectors, boundaries)
4. 4+1 Views
5. Style and Pattern Decisions
6. MVC and Presentation Architecture
7. Package Principles and Dependency Governance
8. Quality Attributes and Scenarios
9. Risks, Trade-offs, and Technical Debt
10. Validation and Evolution Plan
11. Traceability Matrix (Use Cases/Quality Attributes -> Views -> Decisions)

If information is missing, keep the section and mark as TBD.

## Anti-Patterns (MUST avoid)

- Confusing architecture with low-level class design.
- Mixing paradigms without explicit boundaries and rationale.
- Confusing logical layers with physical tiers.
- Omitting interfaces or connector semantics.
- Ignoring quality attributes and non-functional constraints.
- Choosing styles without context and trade-off analysis.
- Leaving dependency cycles unresolved.
- Missing traceability from use cases and quality attributes to architecture decisions.

## Storage Convention

- Persist under: `03 Design/SW Architecture Document.md`

## Validation Checklist (MUST run before finish)

1. Are architecture drivers explicit and prioritized?
2. Are significant decisions and rationale explicit?
3. Are processing, data, and connector elements clearly identified?
4. Are all 4+1 views present with diagrams and constraints?
5. Is architecture kept above class-level implementation detail?
6. Are ADP, SDP, SAP, CRP, CCP, and REP addressed?
7. Are dependency directions/rules and cycle handling explicit?
8. Are style choices justified with context and trade-offs?
9. Are quality attributes prioritized and measurable scenarios defined?
10. Are risks and technical debt identified with mitigation direction?
11. Is traceability explicit from requirements to decisions/views?
12. Is storage path compliant with convention?
13. Are all mandatory MVC subsections present in the required order?
14. Are Model/View/Controller (or Presenter/ViewModel) responsibilities explicit and non-overlapping?
15. Is the Evolution subsection explicit about testability impact and responsibility migration?
16. Is the selected MVC-family variant justified and are non-selected variants explicitly rejected with rationale?
